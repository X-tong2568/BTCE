## 2. 如何拿到B站动态列表

### 2.1 API 端点

```python
# bili_api.py 第15-18行
FEED_SPACE_URL = "https://api.bilibili.com/x/polymer/web-dynamic/v1/feed/space?host_mid={uid}"
```

**这是B站的新版 polymer 接口**（2026年6月，旧版 `api.vc.bilibili.com` 已下线返回404）。

### 2.2 请求方式

```python
# bili_api.py 第51-59行 —— 完整请求代码
req = urllib.request.Request(
    url,
    headers={
        "User-Agent": "Mozilla/5.0 ... Chrome/122.0.0.0 Safari/537.36",
        "Referer": f"https://space.bilibili.com/{uid}/dynamic",
        "Cookie": cookie_str,  # 从 cookies.json 加载
    }
)
```

**关键点**：
- **只需要 Cookie，不需要 WBI 签名**（这是新版 polymer 接口的优势）
- 需要 `User-Agent` 模拟浏览器
- 需要 `Referer` 模拟从空间页发起请求
- 用 `urllib` 同步请求 + `asyncio.to_thread()` 包装成异步（不阻塞事件循环）

### 2.3 响应结构解析

```python
# bili_api.py 第73-108行
data = response.json()
items = data["data"]["items"]  # 新版结构用 items[]，不是旧版的 cards[]

for item in items:
    dyn_id = item["id_str"]          # 动态ID
    dyn_type = item["type"]          # 动态类型
    modules = item["modules"]
    mod_dyn = modules["module_dynamic"]
    
    # 文本内容（可能为 None，纯图动态无 desc）
    desc = mod_dyn.get("desc")
    content = desc.get("text", "") if desc else ""
    
    # 图片列表
    major = mod_dyn.get("major") or {}
    draw_items = major.get("draw", {}).get("items", [])
    for di in draw_items:
        src = di["src"]
        if src.startswith("//"):
            src = "https:" + src  # 补全协议
```

### 2.4 新动态检测逻辑

```python
# monitor.py 第436-449行 —— 差集比对
seen = set(self.history_data.get("seen_dynamics", []))
is_cold = len(seen) == 0  # 冷启动：首次运行不通知

new_dynamics = []
for d in api_list:
    dyn_id = d["dynamic_id"]
    if dyn_id not in seen:
        seen.add(dyn_id)
        if not is_cold:
            new_dynamics.append(d)  # 只通知非冷启动的新动态

# 保留最近500条 seen，防止内存膨胀
self.history_data["seen_dynamics"] = list(seen)[-500:]
```

**冷启动机制**：程序首次运行时，API 返回的所有动态都是"新"的（因为 history 为空），如果全部通知会炸。所以用 `is_cold` 标记静默记录，不触发通知。

## 3. 如何拿到置顶评论

### 3.1 核心方法：Playwright 浏览器抓取

置顶评论无法通过 API 获取，必须用浏览器打开动态页面，从 DOM 中提取。

```python
# render_comment.py 第34-113行 —— 完整抓取流程
async def get_pinned_comment(self, page, dynamic_id):
    # 1. 打开动态页面
    await page.goto(f"https://t.bilibili.com/{dynamic_id}")
    
    # 2. 等待评论区 Web Component 加载
    await page.wait_for_selector("bili-comment-thread-renderer", timeout=15000)
    
    # 3. 滚动加载更多评论
    for _ in range(5):
        await page.evaluate("window.scrollBy(0, 1000)")
        await asyncio.sleep(1)
    
    # 4. 遍历所有评论线程，找置顶标记
    comment_items = await page.query_selector_all("bili-comment-thread-renderer")
    for item in comment_items:
        top_tag = await item.query_selector("i#top")
        if top_tag:
            # ✅ 这就是置顶评论！
            
            # 4a. 提取文字+表情 HTML
            content_element = await item.query_selector("bili-rich-text p#contents")
            if content_element:
                pinned_comment_html = await content_element.inner_html()
            
            # 4b. 提取评论区上传的图片
            pics_renderer = await item.query_selector("bili-comment-pictures-renderer")
            if pics_renderer:
                # 穿透 Shadow DOM 获取图片 URL
                img_src_list = await pics_renderer.evaluate("""
                    (el) => {
                        const imgs = [];
                        const shadow = el.shadowRoot;
                        if (shadow) {
                            shadow.querySelectorAll('img').forEach(img => {
                                let src = img.src;
                                if (src.startsWith('//')) src = 'https:' + src;
                                if (src.includes('@')) src = src.split('@')[0];
                                imgs.push(src);
                            });
                        }
                        return imgs;
                    }
                """)
                comment_images.extend(img_src_list)
            break
    
    return pinned_comment_html.strip(), comment_images
```

### 3.2 B站评论区的 DOM 结构（关键！）

```
动态页面 https://t.bilibili.com/{dynamic_id}
  └─ <bili-comment-renderer id="comment">       ← 评论区容器
       └─ <bili-comment-thread-renderer>         ← 评论线程（Web Component）
            ├─ <i id="top">                      ← 置顶标记（有这个就是置顶评论）
            ├─ <bili-rich-text>
            │   └─ <p id="contents">             ← 评论文字+表情 HTML
            │       ├─ 文字节点
            │       └─ <img alt="[tv_doge]">     ← B站表情（alt 属性是表情名）
            └─ <bili-comment-pictures-renderer>  ← 上传的图片（Shadow DOM 内）
                 └─ #shadow-root
                      └─ <img src="...">
```

**为什么用 `i#top` 而不是位置判断**：B站的置顶评论不一定在第一个位置（页面加载顺序不确定），唯一的可靠标识是 `<i id="top">` 这个元素。

### 3.3 置顶评论变更检测 —— 过滤链路详解

这是整个监控系统最核心的判断逻辑。从 DOM 中拿到的原始 HTML 到最终「是否通知」的布尔值，经过了**两层过滤 + 三条判断路径**。

#### 3.3.0 原始数据长什么样

从 `get_pinned_comment` 拿到的是 `<p id="contents">` 的 `inner_html`，类似：

```html
今天直播很开心<img alt="[tv_doge]" src="//i0.hdslb.com/bfs/emote/xxx.png"><a href="//space.bilibili.com/123" target="_blank">@某用户</a>记得来看下次！
```

这段 HTML 包含：文字节点 + `<img>`（B站表情） + `<a>`（链接/艾特） + 可能的 `<span>` 等格式标签。

#### 3.3.1 第一层过滤：`_clean_html_emojis()` — 表情图片 → 文字占位

```python
# monitor.py 第105-115行
def _clean_html_emojis(self, html_text):
    soup = BeautifulSoup(html_text, 'html.parser')
    for img in soup.find_all('img'):
        alt = img.get('alt', '')
        img.replace_with(alt if alt else "[表情]")  # ← 关键：用 alt 文本替换 img 标签
    return str(soup)
```

| 原始内容 | 处理后 | 说明 |
|---------|--------|------|
| `<img alt="[tv_doge]" src="...">` | `[tv_doge]` | alt 有值 → 用 alt 文本替换 |
| `<img alt="" src="...">` | `[表情]` | alt 为空 → 用兜底占位 `[表情]` |
| `<img src="...">` | `[表情]` | 无 alt 属性 → 用兜底占位 `[表情]` |
| `<a href="...">@某人</a>` | `<a href="...">@某人</a>` | **不处理**，原样保留 |
| 纯文字 `今天直播很开心` | `今天直播很开心` | **不处理**，原样保留 |

**这一步的本质**：把「看不见的图片标签」变成「看得见的文字」，让 emoji 信息能在后续的纯文本比对中存活下来。如果不做这一步，BeautifulSoup 的 `get_text()` 遇到 `<img>` 会直接丢弃（因为 img 是空元素，没有文本子节点），导致「加了表情」这个变化检测不到。

#### 3.3.2 第二层过滤：`extract_text_from_html()` — 剥离所有 HTML 标签，只留纯文本

```python
# render_comment.py 第26-32行
@staticmethod
def extract_text_from_html(html_content: str) -> str:
    if not html_content:
        return ""
    soup = BeautifulSoup(html_content, "html.parser")
    return soup.get_text(strip=True)
```

| 输入（第一层过滤后的 HTML） | 输出（纯文本） | 说明 |
|---------------------------|--------------|------|
| `今天直播很开心[tv_doge]<a href="...">@某用户</a>记得来看` | `今天直播很开心[tv_doge]@某用户记得来看` | 所有 HTML 标签被剥离 |
| `<span style="color:red">重要</span>通知` | `重要通知` | 格式标签被剥离 |
| `<br>` 换行 | （空格） | 换行变空格后 strip 合并 |

**这一步的本质**：把所有「格式层面的差异」全部归一化。UP主可能改了字体颜色、加了粗体、换了链接文字但不换链接目标——这些统统不影响判断。只有**用户肉眼能看到的文字内容**变化才会触发通知。

#### 3.3.3 最终判断：三条路径决定是否通知

```python
# monitor.py 第272-275行 —— 核心判断（不是 render_comment.py 的 detect_comment_change！）
cur_text = self.comment_renderer.extract_text_from_html(self._clean_html_emojis(current_html))
last_text = self.comment_renderer.extract_text_from_html(self._clean_html_emojis(last_html))

should = (
    (is_new_pinned and self.history_data.get("last_pinned_dynamic_id"))  # 路径①
    or (not last_text or cur_text != last_text)                           # 路径②
)
```

**路径①：置顶动态ID被更换了** → **一定通知**

当有人在 QQ 群用 `@Bot 更换置顶 <新动态ID>` 指令换了目标后，`is_new_pinned=True` 且 `last_pinned_dynamic_id` 存在。此时**不管新旧置顶评论内容是否相同，都触发通知**。因为「UP主换了置顶动态」本身就是一个值得通知的事件。且此时用来对比的 `last_html` 是**旧置顶动态**的最后记录，所以新旧不同动态的评论内容几乎必然不同。

**路径②：同一置顶动态下评论内容变了** → **通知**

分两种情况：
- **首次记录**（`not last_text`）：history 里还没有这个动态的评论记录 → 只记录不通知（因为 `not last_text` 为 True 会让 should 为 True，但实际上代码在 `should=True` 之后直接更新 history 然后下次才能对比… 不对，让我重新看）

实际上，`not last_text or cur_text != last_text` 的逻辑是：
- `not last_text` = True → should = True → 会触发通知
- 但往下一行 `_send_notification` 里有邮件/QQ发送，首次确实会通知

不过等等——实际上「首次记录就通知」的场景在正常运行时不会发生，因为程序启动时 history 里已经有上次的记录。只有在 history 文件丢失/清空后的第一次检测才会触发。设计上这也是合理的：history 丢了就是异常情况，能通知就通知。

- **文字变了**（`cur_text != last_text`）：两层过滤后的纯文本不一样 → 通知

#### 3.3.4 评论区上传图片的独立判断

图片变化**不在文字比对路径里**，而是单独判断：

```python
# render_comment.py 第129-132行
if set(current_images) != set(last_images):
    logger.info("🔔 检测到置顶评论图片变化！")
    return True
```

图片比较用的是 **URL 集合对比**（`set()`），所以：
- 增删图片 → 触发
- 换了图片 → 触发（URL 变了）
- 图片顺序变了 → **不触发**（set 无序）
- 同一张图片重新上传 → 触发（B站会生成新 URL）

这个独立判断在 `render_comment.py` 的 `detect_comment_change` 中，而不是在 `monitor.py` 的主判断流程中。但在实际运行时，`monitor.py` 的第275行 `should` 判断实际上**没有调用** `detect_comment_change`，而是自己做了文字比对 + `is_new_pinned` 判断。图片变化是通过 `_send_notification` 中传给邮件/QQ 的对比展示来体现的（新/旧评论 HTML 和图片都传给渲染器，邮件里会展示新旧对比）。

#### 3.3.5 总结：什么被过滤掉，什么留下来

```
原始 inner_html
  │
  ├─ ❌ 过滤掉（不影响判断）：
  │     • <img> 标签本身（替换为 alt 文本或 [表情]）
  │     • <a> 标签的 href 属性（链接目标）
  │     • <span>/<b>/<i>/<font> 等格式标签
  │     • <br> 换行符（转为空格后合并）
  │     • HTML 注释
  │     • 多余的空白字符（strip=True 移除首尾空白）
  │     • 图片排列顺序（set 对比不关心顺序）
  │
  ├─ ✅ 留下来（作为判断依据）：
  │     • 评论文本内容（每个字、标点）
  │     • 表情的 alt 文本（如 [tv_doge]、[星星眼]）
  │     • @用户 的显示名称（@后面的文字，不含链接）
  │     • 评论区上传图片的 URL 列表（集合对比增删换）
  │
  └─ 🔔 触发通知的条件（满足任一）：
       1. 置顶动态ID被更换（is_new_pinned=True + 有旧ID）
       2. 两层过滤后的纯文本与上次不同
       3. 评论区图片 URL 集合与上次不同
```

**实际例子**：假设 UP主把置顶评论从 `"今晚8点直播见[tv_doge]"` 改成 `"<b>今晚8点</b>直播见[tv_doge] <a href='...'>点这里预约</a>"`：

1. 原始 HTML 差异很大（多了 `<b>`、`<a>` 标签）
2. `_clean_html_emojis` 后：`"今晚8点直播见[tv_doge]"` vs `"<b>今晚8点</b>直播见[tv_doge] <a href='...'>点这里预约</a>"`（emojis 已替换，其他标签保留）
3. `extract_text_from_html` 后：`"今晚8点直播见[tv_doge]"` vs `"今晚8点直播见[tv_doge]点这里预约"`
4. 对比结果 → **不同** → 🔔 触发通知

反之，如果 UP主只是把链接换了个目标（href 变了但显示文字不变），或者改了字体颜色/大小：
1. 两层过滤后纯文本完全一样 →
2. 对比结果 → **相同** → ❌ 不通知（避免了因格式微调导致的骚扰推送）

### 3.5 截图耗时拆解 —— B站推送为什么最慢

截图代码中有三处刻意等待的 `sleep`，加上页面导航，完整耗时如下：

```
截图全流程耗时拆解（monitor.py 第304-341行）
═══════════════════════════════════════════════════

  ① goto(wait_until="load")          ████████████  ~3-8秒    ← 最大瓶颈！
  ② wait_for_load_state("networkidle") ████████     ~0-8秒    ← 第二大瓶颈
  ③ asyncio.sleep(1)                 █             1.0秒      ← 等DOM渲染
  ④ hide nav bars (evaluate JS)      ▏            ~0.1秒
  ⑤ asyncio.sleep(0.3)               ▏            0.3秒      ← 等浏览器重排
  ⑥ scroll_into_view                 ▏            ~0.2秒
  ⑦ asyncio.sleep(0.3)               ▏            0.3秒      ← 等懒加载
  ⑧ screenshot()                     ▏            ~0.2秒
                                       ─────────
  刻意sleep合计: 1.0 + 0.3 + 0.3 = 1.6秒  ← 不是瓶颈！
  真正耗时: 页面加载 + networkidle ≈ 5-15秒 ← 这才是瓶颈
```

#### 三处 sleep 各自的作用

| sleep | 位置 | 时长 | 为什么必须等 |
|-------|------|------|-------------|
| `sleep(1)` | 页面 `load` + `networkidle` 之后，操作 DOM 之前 | 1 秒 | B站是 SPA（单页应用），网络请求结束后 JS 还要渲染 DOM。评论区是 Web Component，需要时间 mount 并填充 shadow DOM。**不等的话 `#comment` 可能还不存在** |
| `sleep(0.3)` | 隐藏导航栏（JS 改了几十个元素的 `display:none`）之后 | 0.3 秒 | 浏览器需要时间**重排（reflow）**页面布局。隐藏 fixed 导航栏后，剩余元素的位置会变化，不等的话 `scroll_into_view` 可能滚到错误位置 |
| `sleep(0.3)` | `scroll_into_view_if_needed()` 之后 | 0.3 秒 | B站评论区图片和内容可能**懒加载**（元素滚入视口后才加载）。这个短暂延迟让懒加载的内容来得及出现再截图 |

#### 真正的瓶颈在哪

**刻意 sleep 总共才 1.6 秒**，占截图总耗时的 10~30%。真正的耗时大户是：

1. **`goto(url, wait_until="load")`（3~8秒）**：Playwright 打开 B站动态页，要下载 HTML + JS + CSS + 图片等几十个资源，B站页面又很重（SPA + 大量第三方脚本）

2. **`wait_for_load_state("networkidle")`（0~8秒）**：等网络完全安静。B站页面有长轮询/WebSocket 持续通信，所以设了 8 秒超时兜底，超时了也不阻塞

3. **`new_context(device_scale_factor=2)`（~200-500ms）**：创建独立的高 DPI 浏览器上下文本身有开销

**那为什么不能去掉 sleep 来加速？** 因为 ①② 才是瓶颈，去掉 sleep 省下的 1.6 秒杯水车薪，但去掉后截图成功率会大幅下降——DOM 还没渲染完就截，结果是空白或残缺的截图，发布到 B站就是废图。

#### 三通道推送的时序：text 模式不受截图速度影响

```
_send_notification() 执行时序（monitor.py 第351-398行）
══════════════════════════════════════════════════════

  时刻 0ms    ├─ QQ推送 (text模式) ──▶ ~200ms完成 ✅ 用户秒收
              ├─ 邮件推送 (text模式) ──▶ ~2s完成 ✅
              │
              ├─ 截图 ──▶ 5~15秒 ⏳                    ← 只有 screenshot 模式的通道等这个
              │
  时刻 ~10s   └─ B站发布 ──▶ ~2s (上传图床+发布API) ✅
```

**text 模式（默认）**：QQ 和邮件在截图开始前就已发出，用户几乎实时收到通知。截图只服务于 B站发布通道。

**screenshot 模式**：QQ/邮件也要等截图完成后才能发（因为消息里要嵌入截图），这时延迟就变成 5~15 秒了。

**结论**：B站推送确实是最慢的通道，瓶颈在页面加载（`goto` + `networkidle`），不在那 1.6 秒的 `sleep`。默认配置下 QQ 和邮件走 text 模式，不受截图速度影响，用户在置顶评论变更后 1~2 秒就能收到通知。

### 4.4 状态变更检测逻辑

```python
# live_monitor.py 第159-171行
def detect_status_change(self, current):
    if self.last_live_status is None:
        return False, "initial"  # 首次启动不通知
    
    old = self.last_live_status
    
    # live_status 从 0→1 = 开播, 1→0 = 下播
    if old["live_status"] != current["live_status"]:
        return True, "live_start" if current["live_status"] == 1 else "live_end"
    
    # 标题变化（但 live_status 不变）
    if old.get("title") != current.get("title") and current.get("title"):
        return True, "title_change"
    
    return False, "no_change"
```

**只有这三种变更会触发通知**：`live_start`（开播）、`live_end`（下播）、`title_change`（标题变化）。状态标签（加密/锁房等）不独立触发通知，只作为附加信息展示。

## 5. 为什么需要Cookie

### 5.1 哪些操作需要Cookie

| 操作 | 是否必需Cookie | 原因 |
|------|:---:|------|
| API 获取动态列表 (`polymer feed/space`) | ✅ 必需 | B站需要登录态才能访问用户空间动态 |
| Playwright 打开动态页面 | ✅ 必需 | 未登录会跳转到登录页，无法加载评论区 |
| 直播 API (`get_info` / `getInfoByRoom`) | ⚠️ 建议带 | 部分信息（如封面）不带 Cookie 也能拿，但带 Cookie 更稳定 |
| room_init API | ⚠️ 建议带 | 同上 |
| B站图床上传 (`upload_bfs`) | ✅ 必需 | 需要 `bili_jct`（CSRF token）+ 登录态 |
| B站动态发布 (`create/dyn`) | ✅ 必需 | 需要 `bili_jct` + `DedeUserID` + 登录态 |

### 5.2 Cookie 中的关键字段

```json
// cookies.json 中最重要的几个字段
{
  "SESSDATA": "...",      // 登录会话凭证（核心！过期=全部失效）
  "bili_jct": "...",      // CSRF token（发布动态/上传图片必需）
  "DedeUserID": "...",    // 用户 UID（发布动态时拼 upload_id 用）
  "buvid3": "...",        // 设备标识
  "buvid4": "...",        // 设备标识
}
```

### 5.3 如何获取Cookie

**方法1：自动获取（推荐）**
```bash
python get_cookies.py
```
这个脚本会：
1. 启动 Chromium 浏览器（有头模式，你能看到）
2. 打开 B站登录页
3. 等你在浏览器中扫码登录
4. 登录成功后自动保存所有 Cookie 到 `cookies.json`

```python
# get_cookies.py 第8-37行核心逻辑
async def save_cookies():
    async with async_playwright() as p:
        browser = await p.chromium.launch(headless=False)  # 有头模式
        context = await browser.new_context()
        page = await context.new_page()
        
        await page.goto("https://passport.bilibili.com/login")
        
        # 等待登录成功（跳转到首页），最长等2分钟
        await page.wait_for_url("https://www.bilibili.com/", timeout=120000)
        
        # 保存 cookies
        cookies = await context.cookies()
        Path(COOKIE_FILE).write_text(json.dumps(cookies, indent=2))
```

**方法2：手动导出**
1. 在 Chrome 中登录 B站
2. 按 F12 打开开发者工具 → Application → Cookies → 复制需要的字段
3. 按 `cookies.json` 格式写入

### 5.4 Cookie 有效期与维护

- **SESSDATA 约7天过期**，过期后所有功能失效
- Cookie 过期现象：API 返回 `code=-101`（未登录）、B站图片上传报 `-101`
- 建议：定期（每周）重新运行 `get_cookies.py`

## 8. B站动态自动发布需要什么

当检测到置顶评论变更时，系统可以自动在 B站发布一条图文动态（配截图 + 话题标签 + 跳转链接），让关注你的粉丝也能看到更新提醒。整个发布流程分三步：**截图 → 上传图床 → 组装 JSON 发布**。

### 8.1 发布触发条件

```python
# monitor.py 第387-396行 —— 只在 screenshot 模式 + 开关开启时触发
if AUTO_PUBLISH_ENABLED and BILI_MODE == "screenshot":
    if not screenshot_path:
        screenshot_path = await self._take_pinned_comment_screenshot(dynamic_id)
    if screenshot_path:
        asyncio.create_task(
            auto_publish.publish_dynamic(dynamic_id, screenshot_path,
                                         await self.context.cookies(),
                                         UP_NAME, AUTO_PUBLISH_TOPIC_ID, AUTO_PUBLISH_TOPIC_NAME)
        )
```

**三个必要条件同时满足才会触发**：

| 条件 | 配置位置 | 说明 |
|------|---------|------|
| `AUTO_PUBLISH_ENABLED = True` | `config.py` 第101行 | 总开关，False 则整个功能关闭 |
| `BILI_MODE = "screenshot"` | `config.py` 第98行 | text 模式跳过不发布 |
| 截图文件存在 | 运行时生成 | 截图失败则什么都不发，不抛异常 |

**注意**：发布是用 `asyncio.create_task` 异步触发的，不阻塞 QQ/邮件通知。如果发布失败，不会影响其他两个通道。

### 8.2 前置条件：发布账号要求

| 条件 | 为什么 |
|------|--------|
| **B站账号已实名认证** | 未实名账号调用发布 API 直接返回 `-101` 未登录 |
| **Cookie 含 `bili_jct`** | 作为 CSRF token，URL 参数和 multipart form 两处都要传 |
| **Cookie 含 `DedeUserID`** | 用于拼装 `upload_id`（B站去重标识） |
| **Cookie 含 `SESSDATA`** | 登录会话凭证，过期则 -101 |

### 8.3 第一步：上传截图到 B站图床

```python
# auto_publish.py 第42-91行
BILI_UPLOAD_URL = "https://api.bilibili.com/x/dynamic/feed/draw/upload_bfs"

async def upload_image(file_path, cookies):
    csrf = _csrf_from_cookies(cookies)      # ① 从 Cookie 提取 bili_jct
    cookie_str = _cookie_header(cookies)     # ② Cookie 拼成请求头字符串
    
    img_bytes = Path(file_path).read_bytes()
    form = aiohttp.FormData()                # ③ multipart/form-data 上传
    form.add_field("file_up", img_bytes, filename="screenshot.png", content_type="image/png")
    form.add_field("category", "daily")      # 分类：日常
    form.add_field("biz", "new_dyn")         # 业务：新动态
    form.add_field("csrf", csrf)             # CSRF 令牌（注意：form里也要传！）
    
    headers = {
        "User-Agent": "Mozilla/5.0 ... Chrome/131.0.0.0 Safari/537.36",
        "Referer": "https://t.bilibili.com/",
        "Origin": "https://t.bilibili.com",
        "Cookie": cookie_str,
    }
    # POST → 返回 {code: 0, data: {image_url, image_width, image_height, img_size}}
```

**上传成功返回的数据结构**（后续发布动态时原样填入 `pics` 数组）：

```json
{
  "img_src": "https://i0.hdslb.com/bfs/new_dyn/xxxxx.png",
  "img_width": 2160,
  "img_height": 3840,
  "img_size": 512.3
}
```

**为什么 csrf 在 form 字段和 Cookie 头都要传**：B站后端会交叉校验——Cookie 里的 `bili_jct` 和 multipart form 里的 `csrf` 字段必须一致。

**常见失败原因**：

| 现象 | 原因 | 解决 |
|------|------|------|
| HTTP 412 | 缺少 `User-Agent` 请求头 | 加上模拟浏览器的 UA |
| `code=-101` 账号未登录 | Cookie 过期或账号未实名 | 重新扫码登录；完成实名认证 |
| `code=-400` 或 `-500` | 图片格式/大小不对 | 确认是 PNG，文件能正常读取 |

### 8.4 第二步：组装动态 JSON 并发布

```python
# auto_publish.py 第94-159行 —— publish_dynamic 完整流程
BILI_PUBLISH_URL = "https://api.bilibili.com/x/dynamic/feed/create/dyn"

async def publish_dynamic(dynamic_id, screenshot_path, cookies, up_name, topic_id, topic_name):
    # ── 1) 先上传图片拿到 img_info ──
    img = await upload_image(screenshot_path, cookies)
    
    # ── 2) 拼装 upload_id（B站去重标识）──
    mid = _mid_from_cookies(cookies)                          # DedeUserID
    upload_id = f"{mid}_{int(time.time())}_{random.randint(1000, 9999)}"
    
    # ── 3) 组装发布 JSON ──
    body = {
        "dyn_req": {
            "scene": 2,                                       # 场景：图文动态
            "content": {
                "contents": [
                    {
                        "raw_text": f"【{up_name}】瞳瞳空间更新啦~ {link_url}",
                        "type": 1,                            # type=1 纯文本（B站自动识别URL）
                        "biz_id": ""
                    }
                ]
            },
            "pics": [img],                                    # ④ 上传返回的图片信息
            "topic": {                                        # ⑤ 话题标签
                "id": topic_id,
                "name": topic_name,
                "from_source": "dyn.web.list",
                "from_topic_id": 0
            },
            "option": {
                "up_choose_comment": 0,                       # 不开启精选评论
                "close_comment": 0                            # 不关闭评论区
            },
            "meta": {
                "app_meta": {
                    "from": "create.dynamic.web",             # ⑥ 来源标识（必须）
                    "mobi_app": "web"
                }
            },
            "upload_id": upload_id                            # ⑦ 去重ID
        }
    }
    
    # ── 4) POST 发布 ──
    csrf = _csrf_from_cookies(cookies)
    url = f"{BILI_PUBLISH_URL}?csrf={csrf}&platform=web"     # ⑧ csrf在URL参数
    headers = {
        "Content-Type": "application/json",
        "User-Agent": "Mozilla/5.0 ... Chrome/131.0.0.0 Safari/537.36",
        "Referer": "https://t.bilibili.com/",
        "Origin": "https://t.bilibili.com",
        "Cookie": cookie_str,
    }
    async with session.post(url, json=body, headers=headers) as resp:
        result = await resp.json()
        # 成功: code=0, data.dyn_id_str=新动态ID
```

### 8.5 逐字段解析：链接怎么嵌入的、话题标签怎么带的

#### 链接嵌入：`type: 1` 纯文本 + 完整 URL

```json
"contents": [
    {
        "raw_text": "【星瞳_Official】瞳瞳空间更新啦~ https://t.bilibili.com/1208161183564038147?comment_on=1",
        "type": 1,
        "biz_id": ""
    }
]
```

**为什么用 `type: 1` 而不是 `type: 4` 或 `type: 5`**（`踩坑记录.md` 第2节）：

| type 值 | 发布后效果 | 结论 |
|---------|-----------|------|
| `type: 1` | B站自动识别 URL 为可点击链接，显示为蓝色超链接文字 | ✅ 用这个 |
| `type: 4` | 显示为"投票"卡片 | ❌ 完全错误 |
| `type: 5` | 显示为纯文本"网页链接"，**不可点击** | ❌ 不能用 |

**链接 URL 里 `?comment_on=1` 的作用**：打开动态时自动定位到评论区，用户点进去直接能看到置顶评论，不需要再手动下滑。

**`spm_id_from=333.1387.0.0` 的作用**：B站统计追踪参数，用于标识流量来源，非必需但建议保留。

#### 话题标签：`topic` 结构体

```json
"topic": {
    "id": 66066,                   // 话题ID（数字）
    "name": "小星星的家",            // 话题名称（发布后显示为 #小星星的家#）
    "from_source": "dyn.web.list", // 来源标识
    "from_topic_id": 0             // 从哪个话题跳转来的，0=直接创建
}
```

**怎么找到话题 ID**：在 B站网页版搜索话题名 → 进入话题页 → URL 中的数字就是话题 ID。例如 `https://t.bilibili.com/topic/66066` → ID 是 `66066`。

**配置位置**（`config.py` 第101-103行）：

```python
AUTO_PUBLISH_ENABLED = True   # 总开关
AUTO_PUBLISH_TOPIC_ID = 66066          # 话题ID
AUTO_PUBLISH_TOPIC_NAME = "小星星的家"  # 话题名称
```

#### `upload_id`：B站去重标识

```python
mid = "123456789"                           # 从 Cookie 取 DedeUserID
upload_id = f"{mid}_{int(time.time())}_{random.randint(1000, 9999)}"
# 结果示例: "123456789_1718400000_3847"
```

**作用**：防止短时间内重复发布相同的动态。B站后端用这个 ID 做幂等校验，同一个 `upload_id` 不会创建两条动态。所以每次发布都用 `时间戳_随机数` 保证唯一性。

**如果没有 `upload_id` 或重复**：B站会拒绝发布，返回 `code=-1` 或 `code=-500`。

#### `meta.app_meta.from`：来源标识

```json
"meta": {"app_meta": {"from": "create.dynamic.web", "mobi_app": "web"}}
```

**必须写死为 `"create.dynamic.web"`**。如果写成其他值（如 `"dynamic.web"` 或空字符串），B站会拒绝发布（可能不在错误信息里明说，但 code 不为 0）。

#### `scene: 2`：动态类型

- `scene: 2` = 图文动态（有图片的文本动态）
- `scene: 1` = 纯文本动态
- `scene: 4` = 转发动态

因为我们要带截图，所以用 `2`。

### 8.7 风控注意事项

- **正常使用不会触发风控**：每次置顶评论变更只发一条动态，频率极低（可能几天一次）
- **测试阶段的风险**：短时间内多次调用发布 API 并改变 `content` 格式 → 可能触发实名验证弹窗
- **小号风险**：未实名小号调用上传/发布 API 直接 `-101`
- **建议**：用 UP 主本人已实名的大号 Cookie，生产环境正常触发无风险

## 9. 关键技术细节与踩坑

### 9.1 B站 Shadow DOM 的坑

B站评论区大量使用 Web Components（Shadow DOM），普通 `query_selector` 穿透不了。

```javascript
// ❌ 直接 query_selector 拿不到 Shadow DOM 内的图片
document.querySelectorAll('bili-comment-pictures-renderer img')  // 返回空

// ✅ 必须用 .shadowRoot 穿透
el.shadowRoot.querySelectorAll('img')
```

见 `踩坑记录.md` 第1节 和 `render_comment.py` 第69-89行。

### 9.2 截图坐标系的坑

- `element.getBoundingClientRect()` 返回的坐标与 `page.screenshot(clip=...)` 的坐标系 **不一致**
- 最可靠的方式：直接用 `locator.screenshot()`，让 Playwright 自己算坐标

### 9.3 Python 版本兼容

```python
# ❌ Python 3.10+ 语法，3.9 报错
def func() -> dict | None: ...

# ✅ 兼容 Python 3.8/3.9
from typing import Optional
def func() -> Optional[dict]: ...
```

服务端（Rocky Linux 9.6）自带 Python 3.9，本地 Windows 可能用 3.13。写代码注意用 `Optional`/`Union`。

### 9.4 浏览器内存管理

Playwright Chromium headless 模式内存占用约 200-500MB，长时间运行会缓慢增长。

**对策**（`monitor.py` 第96-103行）：
```python
BROWSER_RESTART_INTERVAL = 10  # 每10轮循环重启浏览器

async def restart_browser_if_needed(self):
    if self.loop_count % BROWSER_RESTART_INTERVAL == 0:
        await self.safe_close_browser()  # 关闭旧实例
        await self.initialize_browser()  # 启动新实例
```

### 9.5 页面跳转导致的执行上下文销毁

B站动态页面加载时会经历多次跳转（重定向），如果在跳转期间执行 `page.evaluate()`，会报 "Execution context was destroyed"。

**对策**（`monitor.py` 第135-149行）：
```python
async def _safe_evaluate(self, page, js, retries=3):
    for attempt in range(retries):
        try:
            return await page.evaluate(js)
        except Exception as e:
            if "Execution context" in str(e) and attempt < retries - 1:
                await asyncio.sleep(1.5)
                await page.wait_for_load_state("load", timeout=5000)
            else:
                raise
```

### 9.6 双通道独立统计

置顶评论（Playwright浏览器）和 API 动态列表（urllib HTTP）是两个完全独立的通道，各自有独立的：
- 成功/失败计数器
- P1 告警阈值（浏览器100次 vs API 10次连续失败）
- P2 告警阈值（浏览器80% vs API 90%成功率）

```python
# config.py 第73-78行
P1_TOTAL_FAILURE_THRESHOLD = 100    # 浏览器：累计失败≥100告警
P2_SUCCESS_RATE_THRESHOLD = 0.8     # 浏览器：成功率<80%告警
API_P1_FAILURE_THRESHOLD = 10       # API：连续失败≥10告警（更快响应）
API_P2_SUCCESS_RATE_THRESHOLD = 0.9 # API：成功率<90%告警（更严格）
```

**为什么 API 阈值更严格**：API 请求比浏览器轻量得多，失败往往是接口问题（下线/限流/封禁），需要更快告警。

### 9.7 推送模式三通道可配

```python
# config.py 第96-98行
QQ_MODE = "text"         # QQ: text=纯文本+图片CQ码, screenshot=截图
EMAIL_MODE = "text"      # 邮件: text=文字+表情+图片, screenshot=截图base64
BILI_MODE = "screenshot" # B站: screenshot=截图发布, text=跳过不发布
```

**text 模式的优势**：不等待截图完成，秒级推送。截图延迟到 screenshot 模式通道，不影响其他通道。
